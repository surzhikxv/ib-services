# Бот: воронка BotHelp → aiogram

Пакет `bot/` — рабочая воронка на aiogram: после `/start` бот ведёт человека по шагам,
а контент каждого шага **дословно** перенесён из BotHelp (тексты Telegram MarkdownV2 с
экранированием как в данных, те же кнопки в том же порядке и раскладке, те же вложения).

Переходы между шагами восстановлены из сырья BotHelp (`button → run_bot → referral`,
служебные action/delay-звенья проматываются) — см. `bot/routing.py`. Платёжные ссылки
задаются отдельно (`bot/links.py`) — их можно вписать позже, бот работает и без них.

## Граф воронки

```
/start → [0] Приветствие
[0] «Мне интересно»  → [7] видео-приветствие
[7] «Продолжить»     → [1] Выбор пакета            «Назад» → [0]
[1] «💚 Базовый»      → [2] Инфо базовый
    «⭐ Стандарт +»    → [3] Инфо стандарт
    «👑 Премиум»       → [4] Инфо премиум            «Назад» → [7]
[2/3/4] «Оплата»      → платёжная ссылка (bot/links.py)   «Назад» → [1]
после оплаты → новый экран доступа: «Войти в канал» / «Перейти в канал» + «Назад»
```

Старые страницы «Оплата прошла» (5/6/8) остаются в сырье и доступны через `/step 5|6|8`
для сверки импортированного контента. В реальном post-payment сценарии webhook отправляет
новый экран: кнопка «Войти в канал» / «Перейти в канал» и кнопка «Назад». Ручного
подтверждающего шага в post-payment сценарии нет: бот выдаёт персональный инвайт
или обрабатывает системный Telegram `chat_join_request` после проверки оплаты.

**Чистый чат:** при переходе на следующий шаг (`/start`, кнопки навигации, страница после
оплаты) бот сначала удаляет сообщения предыдущего шага, затем отправляет новый — в чате виден только актуальный шаг.
Исключение — страницы «Оплата прошла» (5/6/8): после отправки они остаются в чате
даже при возврате к пакетам, чтобы пользователь не потерял доступ и инструкции.
Реализация: `bot.py` запоминает `message_id` показанного шага по чату (`STEP_MESSAGES`) и
стирает их при следующем `send_step(..., track=True)`. Служебные `/all` и `/step` идут с
`track=False` — историю чата не трогают (это сверка контента, а не воронка).

## Трекинг воронки в озеро (clickstream)

Бот пишет каждый шаг по юзеру в озеро (`events`, `source_system="telegram_bot"`),
best-effort через `_emit` — недоступность озера не ломает воронку и ответ Prodamus.
События: `bot_start` (на `/start`), `step_enter` (каждый клик-переход, append-only по
`callback_query.id`), `applied` (legacy/debug-кнопка из старого сырья), `payment` (вебхук Prodamus). Этапы
проставляются по `routing.STAGE_BY_STEP`; повторы и возвраты «Назад» видны как
отдельные строки. Платёжная ссылка не трогается — отдельного `checkout` нет (клик по
URL-кнопке Prodamus боту не виден); сигнал «дошёл до оплаты» = `step_enter` на пакете.

**Атрибуция через deep-link.** Маркетинговые ссылки вида
`t.me/SamodvijenieBot?start=<payload>` несут источник в `payload` (Telegram разрешает
`[A-Za-z0-9_-]`, ≤64). Конвенция: пары `ключ-значение` через `_`, ключ от значения —
первым `-`. Алиасы: `s`=utm_source, `m`=utm_medium, `c`=utm_campaign, `ct`=utm_content,
`t`=utm_term. Пример: `?start=s-ig_m-cpc_c-july`. Payload без валидных пар (напр.
`promo2025`) сохраняется в `Source.code` дословно. Источник привязывается к
`subscriber.source_id` и к событию `bot_start`.

Источник бота пишется как `Source(kind="start_link")`; контент-коннекторы пишут
`kind="utm"`. Натуральный ключ `sources` — пара `(kind, code)`, поэтому при
одинаковом `code` это РАЗНЫЕ строки. Склейка «контент → подписчик» в дашборде —
join по нормализованному `code` (через `normalize_utm`), а не по `source_id`/`kind`.

## Источник контента

Единственный источник правды — сырьё из BotHelp в `raw/bothelp_raw.json` (публичный
эндпоинт, только чтение). Это **данные клиента**, поэтому каталог `raw/` в гит не коммитится
(см. `.gitignore`). Парсер `bot/content.py` читает сырьё как есть — без переэкранирования и
без редактирования текстов/подписей.

```bash
python -m bot.fetch     # выгрузить raw/bothelp_raw.json из BotHelp (POST publicMfa)
```

## Что в каких шагах

28 шагов = **11 контентных** + **17 заглушек** (шаги `action`/`delay` приходят без контента —
по ним логика позже).

- Текст: 11 текстовых сообщений (MarkdownV2).
- Вложения: `video_note` на шаге 1 (реальная ссылка на mp4 — aiogram скачает и зальёт сам)
  и `video` на шаге 7 — видео-приветствие. В публичной выгрузке BotHelp у него нет ссылки
  (`storageFileId=null`), поэтому файл берём локально: `bot/media.py` сопоставляет шаг 7 блок 1
  с `media/welcome.mp4` (каталог/файл настраиваются через `BOT_MEDIA_DIR` / `WELCOME_VIDEO_PATH`,
  см. `.env.example`). Файла нет на месте → аккуратная заглушка «будет добавлено», бот не падает.
- Кнопки: `postback` (обычные) и `web_url`. Старые ссылки «Перейти в канал» — настоящие `t.me`,
  используются как fallback, если авто-инвайты каналов не настроены. «Оплата» — динамические переменные BotHelp `{%payment_N%}`:
  реальный адрес подставляет логика, поэтому пока это кнопка-заглушка с той же подписью.

## Проверка 1:1 — оффлайн, без токена

```bash
python -m bot.preview
```

Печатает по каждому шагу точный текст (в кавычках — видно экранирование), кнопки по рядам и
вложения; в конце сверяет, что отправляемые тексты **побайтово** совпадают с `raw/bothelp_raw.json`.

## Запуск живого бота в Telegram

```bash
export TELEGRAM_BOT_TOKEN=123456:ABC...   # токен у @BotFather
python -m bot.bot
```

Команды в чате:

- `/start` — запустить воронку (шаг 0, дальше по кнопкам);
- `/all` — служебное: прислать подряд контент всех 28 шагов с разделителями `── ШАГ N ──`
  (для сверки контента 1:1 с BotHelp);
- `/step N` — служебное: один шаг по индексу `0..27`.

## Оплата через Prodamus

Магазин один (`<домен>.payform.ru`), секретный ключ один, товаров три (по тарифу).
Настройка в `.env` (секреты в гит не попадают):

```bash
PRODAMUS_DOMAIN=samodvizhenieslapychev.payform.ru
PRODAMUS_SECRET=...            # секретный ключ магазина (Prodamus → Настройки → Интеграции)
PUBLIC_BASE_URL=https://...    # публичный адрес приёма вебхука (домен или туннель), без слэша
PRODAMUS_WEBHOOK_PORT=8081
PAYMENT_RETURN_URL=https://t.me/<бот>   # куда вернуть после оплаты (необязательно)
PRODAMUS_SIGN_LINKS=          # 1 — подписывать ссылку (защита цены); включить после проверки
```

**Исходящее — ссылка.** Когда заданы домен+ключ, кнопка «Оплата» становится рабочей
ссылкой на Prodamus с зашитым `order_id` (`tg<id>-<тариф>-<время>`) и ценой тарифа —
один тап ведёт на оплату. Цены/наименования товаров — в `bot/payments.py` (`TARIFFS`).

**Входящее — вебхук.** После оплаты Prodamus шлёт `POST` на `PUBLIC_BASE_URL/prodamus`
с заголовком `Sign` (HMAC-SHA256 всех полей ключом магазина). Бот поднимает приём
рядом с поллингом, **проверяет подпись** (`bot/payments.py`), разбирает `order_id` →
`tg_id`+тариф, пишет оплату в озеро (`payments`, идемпотентно) и шлёт человеку новый
экран доступа в канал.

**Доступ в канал** (`bot/channel.py`): лучший UX включается, когда бот — админ канала и
задан его `chat_id` (`CHANNEL_BASIC_ID` / `CHANNEL_STANDARD_ID` / `CHANNEL_PREMIUM_ID`).
После оплаты бот создаёт одноразовый инвайт и показывает кнопку «Войти в канал» плюс «Назад».
Если Telegram присылает системный `chat_join_request`, бот проверяет успешную оплату
тарифа в БД и подтверждает доступ автоматически. Ручного подтверждающего шага в
post-payment экране нет.

**Локальная проверка вебхука** (без реальной оплаты), при запущенном боте:

```bash
python - <<'PY'
from pathlib import Path; from dotenv import load_dotenv; load_dotenv(Path.cwd()/".env")
import httpx; from bot import payments; from bot.webhook import _listify, _parse_nested
items=[("order_id",payments.make_order_id(123,"premium")),("payment_status","success"),
       ("sum","2990"),("products[0][name]","Премиум пакет"),("products[0][price]","2990"),
       ("products[0][quantity]","1")]
data=_listify(_parse_nested(items))
httpx.post("http://127.0.0.1:8081/prodamus", data=dict(items),
           headers={"Sign":payments.sign(data)}, trust_env=False)
PY
```

**Тест воронки без оплаты:** `BOT_SIMULATE_PAYMENT=1 python -m bot.bot` — нажатие «Оплата»
сразу ведёт на страницу «оплачено» (доступно, только когда Prodamus не настроен).

### Что нужно, чтобы оплата заработала «вживую»

1. **Полный секретный ключ** в `PRODAMUS_SECRET` (на скрине поле было обрезано — сверьте).
2. **Публичный HTTPS** на порт вебхука: туннель для теста —
   `cloudflared tunnel --url http://localhost:8081` (или `ngrok http 8081`), его адрес → в
   `PUBLIC_BASE_URL`. Тогда ссылка сама понесёт `urlNotification` на наш приём.
   (Либо указать наш `…/prodamus` в настройках уведомлений магазина Prodamus.)
3. Добавить бота **админом** в каналы и вписать `CHANNEL_*_ID` — это включает лучший UX:
   после оплаты пользователь получает персональную кнопку входа без ручного шага.

## Файлы

```
bot/
  content.py   парсер сырья BotHelp → шаги/блоки/кнопки (как есть, без правок текста)
  routing.py   граф переходов воронки (button → run_bot → referral, через action/delay)
  payments.py  Prodamus: платёжная ссылка с order_id + подпись (HMAC) + разбор order_id
  webhook.py   приём вебхука Prodamus (aiohttp): разбор формы, сверка подписи → on_paid
  channel.py   выдача доступа в канал оплатившим (одноразовый инвайт, если бот — админ)
  links.py     запасные статические платёжные ссылки + режим симуляции оплаты
  render.py    чистая отрисовка: раскладка кнопок по рядам, тип кнопки (ссылка/заглушка)
  preview.py   оффлайн-превью контента 1:1 + побайтовая сверка с сырьём (без токена)
  bot.py       живой aiogram-бот: воронка по /start, приём оплат, служебные /all и /step N
  fetch.py     выгрузка raw/bothelp_raw.json из BotHelp
tests/test_bot_content.py    контракт контента: 28 шагов, дословность текстов, кнопки, вложения
tests/test_bot_routing.py    контракт воронки: переходы кнопок восстановлены верно
tests/test_bot_payments.py   контракт Prodamus: подпись, ссылка, order_id, разбор вебхука
```
